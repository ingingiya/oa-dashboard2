BEFORE YOU START

클로드로 직접 코딩하는 것과 클로드 코드는 왜 다른가요?

같은 AI인데 — 결과가 완전히 다릅니다

실제 경험:
클로드(브라우저)로 코드를 직접 복사·수정하며 만들었을 때 → 4주
클로드 코드로 동일한 작업을 했을 때 → 2주

같은 사람, 같은 AI, 같은 목표 — 차이는 도구였습니다.

무엇이 다른가요?

상황클로드 (브라우저)클로드 코드
코드 수정 Claude가 코드 생성 → 내가 복사 → 파일 열고 붙여넣기 → 저장 Claude가 파일을 직접 열고 수정·저장까지 자동
에러 발생 시 에러 복사 → 브라우저에 붙여넣기 → 수정 코드 받기 → 다시 복사 → 붙여넣기 Claude가 에러를 직접 보고 즉시 수정
여러 파일 수정 파일마다 열고 복사하고 붙여넣기 반복 한 번에 여러 파일 동시 수정
배포 터미널에 명령어 직접 입력 "배포해줘" 한 마디로 자동
맥락 유지 대화창 넘어가면 처음부터 다시 설명 CLAUDE.md로 프로젝트 항상 기억
쉽게 말하면:
클로드(브라우저)는 설계도를 그려주는 건축가입니다.
클로드 코드는 설계부터 시공까지 직접 하는 건축가입니다.

브라우저 클로드가 나쁜 게 아닙니다 — 기획·아이디어 단계엔 여전히 최고입니다.
실제로 만들 때는 클로드 코드가 압도적으로 빠릅니다.

언제 뭘 쓰나요?

클로드 (브라우저) 쓸 때클로드 코드 쓸 때
어떤 기능을 만들지 기획할 때실제로 코드를 만들고 수정할 때
샘플 UI 미리 확인할 때에러를 고칠 때
아이디어 검토, 방향 잡기배포할 때
문서 작성, 분석, 번역자동화 설정할 때

CHAPTER 01

Claude Code란?

AI가 코드를 대신 써줍니다

Claude Code는 Anthropic이 만든 AI 코딩 도우미입니다. 터미널(검은 화면)에서 실행하면, Claude가 파일을 직접 열고 수정하고 배포까지 해줍니다.

여러분은 한국어로 말만 하면 됩니다.

예시: "무신사 순위 탭 만들어줘. 순위/가격/할인율 보이게 해줘." → Claude가 알아서 코드를 씁니다.
구분ChatGPT / Claude.aiClaude Code
파일 수정복사해서 직접 붙여넣기자동으로 파일 수정
명령 실행불가능직접 실행
배포불가능Vercel에 직접 배포
CHAPTER 02

실제 작업 흐름

이 순서로 진행하면 실패가 적습니다

1
Claude.ai(브라우저)에서 기획 + 샘플 받기

원하는 기능을 설명하고 HTML 샘플 미리보기를 받아봅니다. 코드 없이 방향만 잡는 단계입니다.

"무신사 스타일 상품 순위 대시보드 만들고 싶어.
탭별로 플랫폼 나누고, 순위/가격/할인율 보이면 좋겠어.
어떻게 만들지 HTML 샘플로 보여줘."
2
Claude Code에서 실제로 구현

받은 샘플을 Claude Code에 전달해서 실제 프로젝트에 적용합니다.

"이 샘플 기반으로 Next.js 프로젝트에 적용해줘.
무신사 API 연동이랑 Supabase 저장도 추가해줘."
3
기능 하나씩 추가하며 제작

한 번에 다 만들려 하지 말고, 하나씩 확인하면서 진행합니다.

1. "기본 레이아웃이랑 탭 구조 먼저 만들어줘"
2. "상품 리스트 테이블 추가해줘"
3. "GitHub Actions로 매일 자동 수집되게 해줘"
4. "배포해줘"
Claude.ai = 기획, 아이디어, 샘플 (브라우저)    Claude Code = 실제 구현, 배포 (터미널)

import asyncio
from playwright.async_api import async_playwright

async def scrape(url):
    async with async_playwright() as pw:
        browser = await pw.chromium.launch(
            headless=True,
            args=["--no-sandbox", "--disable-blink-features=AutomationControlled"]
        )
        context = await browser.new_context(
            user_agent="Mozilla/5.0 (Macintosh; Intel Mac OS X 10_15_7)..."
        )
        await context.add_init_script(
            "Object.defineProperty(navigator, 'webdriver', {get: () => undefined})"
        )
        page = await context.new_page()
        await page.goto(url, wait_until="networkidle")
        price = await page.text_content(".price-value")
        await browser.close()
        return price

① Supabase에 기준값 저장

② 대시보드가 기준을 읽어 자동 분류
대시보드가 Supabase에서 기준값을 읽어 소재마다 "잘나온/전환/트래픽" 뱃지를 자동으로 붙입니다.

③ Figma MCP 연동
"재제작" 버튼을 누르면 해당 소재의 Figma 파일로 바로 이동합니다. 팀원 모두 같은 Figma 파일에서 수정 가능하고, Claude Code가 변경사항을 대시보드에 자동 반영합니다.

④ Claude에게 말 한마디로 기준 변경
Supabase에 기준값이 저장되어 있기 때문에 코드를 다시 빌드할 필요 없이
"잘나온 기준을 80점으로 올려줘" 라고 Claude에게 말하면 DB 값만 바꿔서 즉시 반영됩니다.

기능 설명 + 어떤 데이터를 보여줄지 + 어디에 저장할지 + 어떻게 보이면 좋은지

Vercel에 배포하면 https://내프로젝트.vercel.app 주소가 생깁니다.

bun dev 없이 수정 확인하려는 경우
코드를 수정해도 bun dev가 실행 중이어야 localhost:3000에서 확인할 수 있습니다.
→ 터미널에 bun dev가 돌아가고 있는지 먼저 확인하세요.
4
에러 메시지를 그냥 닫기
빨간 에러 화면이 뜨면 겁먹고 닫지 마세요. 에러 메시지 전체를 복사해서 Claude에게 보여주면 대부분 바로 해결됩니다.

5
Vercel에는 반영이 안 되는 경우
로컬에서 수정하고 bun dev에서 잘 보인다고 끝이 아닙니다. GitHub에 push해야 Vercel에 자동 배포됩니다.
→ git add . → git commit → git push 순서 잊지 마세요.
